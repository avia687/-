# בדיקות לפני דיפלוי — הקרון 🥪

סקירה סטטית של הפרויקט לפני דיפלוי ל-Railway. **סטטית** אומרת: קוראים קוד וקונפיגורציה בלבד. לא מריצים `npm install`, לא `npm run build`, לא מרימים שרת. המטרה היא לתפוס את התקלות שמפילות דיפלוי או שוברות את האתר בפרודקשן — לפני שהן קורות.

## רקע על הפרויקט (למה הבדיקות האלה)

- מונו-רפו: `client/` (React + Vite) ו-`server/` (Express). אין node_modules משותף — לכל חלק package.json משלו.
- Railway בונה עם `npm run build` (בשורש) ומריץ עם `npm start` (בשורש). אם הסקריפטים האלה חסרים או שבורים — הדיפלוי נכשל.
- בפרודקשן השרת מגיש את `client/dist` **רק כאשר** `NODE_ENV=production`. בלי זה האתר יעלה אבל יציג דף ריק.
- הלקוח קורא ל-API בנתיבים יחסיים (`/api/...`) ומסתמך על כך שהשרת והלקוח באותו דומיין בפרודקשן.
- וואטסאפ נשלח דרך Twilio. אם משתני ה-Twilio חסרים הקוד מדלג בשקט — האתר עובד אבל הטבחים לא מקבלים הזמנות. זו תקלה שקטה ומסוכנת במיוחד.

## הבדיקות

עבור על כולן, גם אם נמצאה תקלה מוקדם — הדוח צריך לשקף את התמונה המלאה.

### 1. קונפיגורציית Railway ↔ סקריפטים

קרא את `railway.json` ואת `package.json` בשורש. ודא ש:
- `buildCommand` ו-`startCommand` שב-railway.json קיימים כסקריפטים ב-package.json של השורש.
- סקריפט ה-`start` מצביע על קובץ שקיים בפועל (כיום `node server/index.js` — ודא שהנתיב נכון).
- סקריפט ה-`build` מתקין תלויות לשני החלקים ובונה את הלקוח (`vite build`). אם הוא בונה רק חלק אחד — הדיפלוי "יצליח" אבל משהו יחסר.

### 2. הגשת הלקוח בפרודקשן

קרא את `server/index.js` וודא:
- קיים בלוק שמגיש קבצים סטטיים מ-`client/dist` (בדוק שהנתיב היחסי מ-server אכן נכון).
- קיים נתיב catch-all (`app.get('*', ...)`) שמחזיר את `index.html` — בלעדיו ניווט ישיר לכתובת כמו `/chef` יחזיר 404.
- **סדר הנתיבים**: כל נתיבי ה-API חייבים להירשם לפני ה-catch-all. נתיב API שנוסף אחריו לא יגיע לעולם.

### 3. תלויות (הכשל הקלאסי של "אצלי זה עובד")

סרוק את כל ה-`require()` בקבצי `server/**/*.js` ואת כל ה-`import` בקבצי `client/src/**`. עבור כל חבילה חיצונית (לא נתיב יחסי ולא מודול מובנה של Node כמו `path`):
- ודא שהיא מופיעה ב-`dependencies` של ה-package.json המתאים (server או client).
- בצד השרת — חבילה שנמצאת רק ב-`devDependencies` היא תקלה: Railway מתקין בפרודקשן בלי devDependencies, והשרת יקרוס בעלייה.
- בצד הלקוח — כלי build כמו vite ב-devDependencies זה תקין; ספריות שנכנסות לקוד עצמו (react, react-router-dom) חייבות להיות ב-dependencies.

### 4. משתני סביבה

אסוף כל שימוש ב-`process.env.*` בקוד השרת. השווה לרשימה ב-`server/.env.example`:
- משתנה שהקוד משתמש בו ולא מתועד ב-.env.example — הוסף אותו (עם ערך דוגמה, לא ערך אמיתי).
- אי אפשר לבדוק מכאן מה מוגדר ב-Railway עצמו, ולכן הדוח חייב לכלול רשימה מפורשת של המשתנים שצריך לוודא ידנית בדשבורד של Railway. כיום: `NODE_ENV=production`, `TWILIO_ACCOUNT_SID`, `TWILIO_AUTH_TOKEN`, `TWILIO_WHATSAPP_FROM`. (`PORT` מסופק אוטומטית על ידי Railway.)
- זכור: בלי משתני Twilio האתר עולה אבל הודעות וואטסאפ לא נשלחות — בלי שום שגיאה נראית לעין. ציין זאת בדוח.

### 5. סודות

- ודא ש-`.env` (גם בשורש וגם ב-server) מופיע ב-`.gitignore`.
- חפש בקוד מחרוזות שנראות כמו סודות אמיתיים: SID של Twilio (`AC` ועוד 32 תווים הקסדצימליים), טוקנים, סיסמאות. ערכי דוגמה ב-.env.example (כמו `ACxxx...`) תקינים.
- אם נמצא סוד אמיתי בקוד — זו תקלה חוסמת. דווח, אל תנסה "לתקן" לבד (הסוד כבר נחשף בהיסטוריית גיט והמשתמש צריך להחליף אותו אצל הספק).

### 6. כתובות קשיחות בלקוח

חפש ב-`client/src/` קריאות `fetch` או כתובות API עם `localhost` או כתובת מלאה (`http://...`). קריאות API חייבות להיות בנתיב יחסי (`/api/...`) כדי לעבוד בפרודקשן. שים לב: קישורי `wa.me` (פתיחת וואטסאפ) הם קישורים חיצוניים לגיטימיים — לא תקלה.

### 7. נתיבי לקוח מול השרת

השווה את הנתיבים שמוגדרים ב-`client/src/App.jsx` (כיום `/` ו-`/chef`) למנגנון ההגשה בשרת. כל נתיב לקוח חדש מכוסה אוטומטית על ידי ה-catch-all — אבל רק אם ה-catch-all קיים ותקין (בדיקה 2).

## מה מתקנים ומה רק מדווחים

**תקן אוטומטית** (ואז דווח מה תוקן) — תיקונים מכניים שאין בהם שיקול דעת:
- חבילה בשימוש שחסרה ב-dependencies → הוסף אותה ל-package.json המתאים (גרסה עדכנית סבירה עם `^`), וציין בדוח שצריך להריץ `npm install` מקומית לפני קומיט של package-lock.
- חבילת שרת שיושבת ב-devDependencies במקום ב-dependencies → העבר.
- משתנה סביבה לא מתועד → הוסף ל-.env.example עם ערך דוגמה.
- `.env` שלא ב-gitignore → הוסף שורה.

**דווח בלבד, אל תשנה** — כל מה שדורש החלטה של בן אדם:
- סקריפט build/start שבור או חסר (התיקון תלוי בכוונה — אל תנחש).
- בעיות סדר נתיבים או הגשה ב-server/index.js (שינוי לוגיקת שרת).
- סוד אמיתי בקוד.
- כתובת קשיחה בלקוח (אולי יש סיבה — שאל).
- מגבלות ארכיטקטורה ידועות. החשובה שבהן: **ההזמנות נשמרות בזיכרון בלבד ונמחקות בכל דיפלוי** — תזכורת קבועה בדוח (אזהרה, לא חסימה), כדי שדיפלוי לא יתבצע באמצע משמרת עמוסה.

כתוב את הדוח בעברית, תמיד באותו מבנה, כדי שאפשר יהיה לסרוק אותו בשניות:

```
# 🚀 בדיקת מוכנות לדיפלוי — הקרון

## שורה תחתונה: ✅ מוכן לדיפלוי / ❌ לא מוכן — יש תקלות חוסמות

## ❌ תקלות חוסמות (אם יש)
[מה, איפה (קובץ:שורה), ולמה זה יפיל את הדיפלוי או ישבור את האתר]

## 🔧 תוקן אוטומטית (אם יש)
[מה שונה ובאיזה קובץ]

## ⚠️ אזהרות ותזכורות
[כולל תמיד: רשימת משתני הסביבה לאימות ידני ב-Railway, ותזכורת אחסון-בזיכרון]

## ✅ עבר בדיקה
[שורה אחת לכל בדיקה שעברה — בלי להאריך]
```

חסימה (❌) היא רק תקלה שתפיל את הדיפלוי או תשבור פונקציונליות בפרודקשן. כל השאר — אזהרה. אל תנפח אזהרות לחסימות: דוח שזועק "לא מוכן" על כל קטנה מאבד את האמון של מי שקורא אותו.
